From code review to continuous authorization.

Code review was built as a single gate: a change arrives, a person reads it, and a decision gets made once. That design assumed a fairly steady trickle of human contributions. Feed it a contributor that never sleeps and the single gate starts to resemble the thing network security spent the last decade dismantling: a perimeter that means less every year.

One good decision does not scale into a thousand good decisions

Peer review always did two jobs at once: catching mistakes, and establishing that a specific accountable person actually looked. Both jobs assumed a workload a person could keep up with. An agent capable of a steady stream of individually reasonable changes strains the second job first. The gate still fires. Someone, or something, still clicks approve. But "someone looked at this" quietly stops describing what happened, because looking at forty things with the attention meant for four is not looking at all, it is skimming with a rubber stamp in hand.

The perimeter model, and why it keeps failing the same way

Enterprise networks spent years assuming that anything already inside the firewall could be trusted by default, the same way engineering teams have long assumed that anything a known contributor proposes deserves an ordinary amount of scrutiny. Both assumptions broke for the same underlying reason: a boundary check performed once at the door tells you nothing about what happens after someone walks through it. The security industry's answer was zero trust: authenticate and authorize every request on its own merits, continuously, regardless of where it originated or what was true five minutes earlier. Engineering workflows are quietly adopting the identical shape. A change is not "reviewed" as a single past-tense event so much as continuously authorized, by role separation, by an automated gate, by a scope check that runs every time, with a human still in the loop for the narrow slice of decisions that still need one.

Trust that accrues instead of trust that is assumed

The other half of the shift is progressive rather than binary. Instead of granting an agent a fixed scope on day one and hoping it holds up, capability expands as evidence accumulates: a narrow scope proves itself reliable, and only then does a slightly wider one get authorized, each step earned by what the previous step actually demonstrated. It is the same logic that replaced flipping a feature on for every user simultaneously, and the same logic behind staged permission grants that replaced handing out broad access on the assumption that it would probably be fine. Earn trust in increments small enough that any single mistake stays cheap, and always keep the door open to roll one increment back.

The reframe worth keeping

Treated as a workflow question, this feels like an ever-growing checklist. Treated as a security question, it becomes a lot more tractable, because security already learned this lesson once. Oversight stops being something completed before an agent starts working and becomes a property the system maintains for as long as the agent keeps working, the same way a zero trust network never considers a device permanently cleared. Teams that already think this way about people tend to adapt fastest, because extending the same discipline to a new kind of contributor turns out to be the easy part.
