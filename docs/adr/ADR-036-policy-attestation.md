# ADR-036: Policy-Pack Manifest and Disclosure Attestation

**Status:** Accepted  
**Date:** 2026-07-03  
**Milestone:** 6 (Self-governance hardening)  
**Builds on:** ADR-017 (knowledge packet signing), ADR-024 (capability promotion gate), ADR-025 (self-application evidence), ADR-033 (repo snapshot)

## Context

Governed Remediation's earlier phases built the deterministic detection kernel, the near-miss
widener with a human-only promotion path, and the armed metadata-only remediator. What the
repository still could not do was state, in one verifiable place, which governance policy it
enforces and how it discloses AI participation. That posture lived as prose spread across
`AGENTS.md` and `README.md`, and the active policy values (the attribution denylists, the
capability flags and their defaults, the protected paths, and the enforced gate set) were
implicit in code with no machine-checkable manifest an owner or an adopter could read or verify.

Two constraints shaped the answer. First, the attribution denylists are hardcoded in the
detector libraries on purpose: CI restores those files from the base branch before the gates
run, so a pull request cannot weaken the gate that judges it. Turning the denylists into an
external data file that the detectors read back from would move policy into PR-editable data and
dissolve that base-pinned trust boundary. Second, the release-evidence generator (ADR-025)
already showed the pattern for a generated artifact that cannot drift into fiction, but its
freshness is not gated in CI, so nothing catches it going stale.

## Decision

Add a single content-addressed **policy attestation**, generated by
`scripts/build-policy-attestation.mjs` (backed by the pure builder
`scripts/lib/policy-manifest.mjs`) and committed at `.modonome/policy-attestation.json`.

The manifest is a **disclosure generated from the authoritative sources, never a source of truth
the detectors read back from**. It fingerprints the four detector libraries by content hash
rather than importing them, reads the shared capability-flag registry and the config for
capability defaults and the arming posture, derives the enforced gate set from the actual
`verify` script, and carries the architectural-level AI-participation statement together with a
section-level fingerprint of the `README.md` and `AGENTS.md` prose that states it. Because it
reads and never writes policy, publishing or verifying a pack cannot weaken the base-pinned
boundary.

The body is serialized with RFC 8785 canonical JSON and addressed by a `sha256`
`content_digest`, so only a real policy change moves the digest. Signing is optional and off by
default: when `MODONOME_SIGNING_KEY` is present at write time, an Ed25519 signature envelope
(ADR-017) is attached over the domain-separated body, using a domain tag distinct from the
knowledge-packet tag so a signature cannot be replayed across artifact types. The body and its
digest are identical whether or not the file is signed.

A `--check` mode rebuilds the manifest from live policy and fails on either a stale digest or a
hand-edited file whose recorded digest no longer matches its body. It is wired into the `verify`
chain and into CI in the pre-base-checkout slot, beside the snapshot and promotion-safety gates,
for the same reason: it must judge the pull request's own policy files and committed artifact,
not the base copy that the later checkout restores. This closes, for the attestation, the
freshness gap that the release-evidence file still has.

## Why a disclosure, not a gate

The attestation attests; it does not authorize. `--check` proves the manifest is current, so a
pull request cannot weaken policy while a stale attestation still claims the old posture. It does
not, and is not meant to, prevent a policy change: a change to a detector or a capability default
moves the digest and must be committed with a refreshed attestation, where it is visible in
review. The mechanisms that actually prevent unreviewed weakening are the base-pinned detectors,
the determinism boundary, and CODEOWNERS on the protected paths. Reading the attestation as an
access-control gate would misplace trust; its value is transparency and tamper-evidence, which
content-addressing and the optional signature provide.

## Consequences

- An owner or an adopter can read one file, or run `modonome attest --show`, to see the enforced
  denylists, capability defaults, protected paths, gate set, and arming posture, and can verify
  with `--check` that it matches the live repository.
- The disclosed gate set derives from the `verify` script, so it cannot drift from what the
  project runs; adding or removing a gate changes the attestation, which `--check` then requires
  to be refreshed.
- The capability-flag list is now shared by the promotion gate and the attestation
  (`scripts/lib/capability-flags.mjs`), so the disclosed capability set and the gated one stay
  identical.
- The artifact is the portable pack format an adopter can vendor and diff. Cross-repo adoption
  tooling and default-on signing are deferred; this ADR adds neither a config lever nor a network
  path.
